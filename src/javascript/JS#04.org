JavaScript es un lenguaje vivo que está añadiendo nuevas funcionalidades constantemente. Como desarrollador de JavaScript, es importante entender el proceso subyacente que se necesita para tomar una nueva característica y transformarla de una simple idea a parte de la especificación oficial del lenguaje. Para hacerlo, cubriremos tres temas: Ecma, EcmaScript y el TC39.

Primero, volvamos a 1995. El clásico de culto Heavy Weights estaba en los cines, Nicolas Cage ganó un Óscar y los sitios web típicamente se veían más o menos así [[https://berkshirehathaway.com/][Berkshire Hathaway Inc.]] Lo más probable es que vieras ese sitio web con [[https://en.wikipedia.org/wiki/Netscape_Navigator][Netscape Navigator]].. En ese momento, Netscape Navigator era el navegador web más popular con casi un 80% del mercado. El fundador de Netscape, la empresa detrás de Netscape Navigator, era [[https://x.com/pmarca][Marc Andreessen]]. Él tenía una visión para el futuro de la web que iba más allá de ser solo una forma de compartir y distribuir documentos. Imaginaba una plataforma más dinámica con interactividad del lado del cliente: una especie de “lenguaje pegamento” que fuera fácil de usar tanto para diseñadores como para desarrolladores. Aquí es donde entra [[https://x.com/BrendanEich][Brendan Eich]].

Brendan fue reclutado por Netscape con el objetivo de incorporar el lenguaje de programación Scheme en Netscape Navigator. Sin embargo, antes de que pudiera comenzar, Netscape colaboró con Sun Microsystems para hacer que Java, su emergente lenguaje de programación, estuviera disponible en el navegador. Ahora bien, si Java ya era un lenguaje adecuado, ¿por qué contratar a Brendan para crear otro? Recordando el objetivo de Netscape, querían “un lenguaje de scripting lo suficientemente simple para que diseñadores y aficionados pudieran usarlo”; lamentablemente, Java no cumplía con eso. A partir de ahí surgió la idea de que Java podría ser usado por “profesionales” y este nuevo lenguaje “Mocha” (el nombre inicial de JavaScript) sería usado por todos los demás. Debido a esta colaboración entre lenguajes, Netscape decidió que Mocha debía complementar a Java y tener una sintaxis relativamente similar.

A partir de ahí, cuenta la leyenda que en solo 10 días Brendan creó la primera versión de Mocha. Esta versión tenía algo de la funcionalidad de Scheme, la orientación a objetos de SmallTalk y, debido a la colaboración, la sintaxis de Java. Eventualmente, el nombre Mocha cambió a LiveScript y después LiveScript cambió a JavaScript como una estrategia de marketing para aprovechar el auge de Java. En este punto, JavaScript se promocionaba como un lenguaje de scripting para el navegador, accesible tanto para aficionados como para diseñadores, mientras que Java era la herramienta profesional para construir componentes web avanzados.

Ahora, es importante entender el contexto en el que ocurrieron estos eventos. Además de que Nicolas Cage ganó un Óscar, Microsoft también estaba trabajando en Internet Explorer. Debido a que JavaScript cambió fundamentalmente la experiencia del usuario en la web, si eras un navegador competidor, y dado que no existía una especificación de JavaScript, no tenías otra opción que crear tu propia implementación. Como muestra la historia, eso fue exactamente lo que hizo Microsoft y la llamó JScript.

Esto llevó a un problema bastante famoso en la historia de Internet. JScript cumplía el mismo propósito que JavaScript, pero su implementación era diferente. Esto significaba que no podías crear un sitio web y esperar que funcionara tanto en Internet Explorer como en Netscape Navigator. De hecho, las implementaciones eran tan distintas que las insignias “Best viewed in Netscape” y “Best viewed in Internet Explorer” se volvieron comunes para muchas compañías que no podían permitirse desarrollar para ambas plataformas. Finalmente, aquí es donde entra Ecma.

Ecma International es “una asociación industrial fundada en 1961, dedicada a la estandarización de sistemas de información y comunicación”. En noviembre de 1996, Netscape presentó JavaScript ante Ecma para crear una especificación estándar. Al hacerlo, permitió que otros implementadores tuvieran voz en la evolución del lenguaje y, idealmente, ayudó a mantener consistentes las implementaciones entre navegadores.

Bajo Ecma, cada nueva especificación viene con un estándar y un comité. En el caso de JavaScript, el estándar es ECMA-262 y el comité que trabaja en este estándar es el TC39. Si buscas el estándar ECMA-262, notarás que el término “JavaScript” nunca se usa. En su lugar, utilizan el término “EcmaScript” para referirse al lenguaje oficial. La razón es que Oracle posee la marca registrada del término “JavaScript”. Para evitar problemas legales, Ecma decidió usar el término EcmaScript. En el mundo real, ECMAScript se usa para referirse al estándar oficial, ECMA-262, mientras que JavaScript se usa para hablar del lenguaje en la práctica. Como se mencionó antes, el comité que supervisa la evolución del estándar Ecma262 es el TC39, que significa Technical Committee 39. El TC39 está compuesto por “miembros” que suelen ser proveedores de navegadores y grandes compañías que han invertido fuertemente en la web, como Facebook y PayPal. Para asistir a las reuniones, los “miembros” (nuevamente, grandes compañías y proveedores de navegadores) envían “delegados” para representar a dicha empresa o navegador. Son estos delegados quienes crean, aprueban o rechazan propuestas de lenguaje.

Cuando se crea una nueva propuesta, esta debe pasar por ciertas etapas antes de convertirse en parte de la especificación oficial. Es importante recordar que para que cualquier propuesta avance de una etapa a otra, debe haber consenso entre el TC39. Esto significa que una gran mayoría debe estar de acuerdo mientras que nadie debe estar en desacuerdo lo suficientemente fuerte como para vetar la propuesta.

Cada nueva propuesta comienza en la Etapa 0. Esta etapa se llama la etapa de “Borrador inicial” (Straw man). Las propuestas en Etapa 0 son “propuestas que se planea presentar al comité por un responsable del TC39 o que ya se han presentado al comité y no han sido rechazadas definitivamente, pero aún no cumplen con los criterios para llegar a la etapa 1”. Así que el único requisito para convertirse en una propuesta de Etapa 0 es que el documento haya sido revisado en una reunión del TC39. Es importante señalar que usar una característica de Etapa 0 en tu código está bien, pero incluso si llega a formar parte de la especificación oficial, casi con certeza pasará por algunas iteraciones antes de eso.

La siguiente etapa en la madurez de una propuesta es la Etapa 1. Para progresar a la Etapa 1, debe identificarse un “responsable” oficial que sea parte del TC39 y esté a cargo de la propuesta. Además, la propuesta necesita describir el problema que resuelve, tener ejemplos ilustrativos de uso, una API de alto nivel e identificar posibles preocupaciones y desafíos de implementación. Al aceptar una propuesta en la Etapa 1, el comité indica que está dispuesto a dedicar recursos para analizarla más a fondo.

La siguiente etapa es la Etapa 2. En este punto, es muy probable que la característica eventualmente forme parte de la especificación oficial. Para llegar a la Etapa 2, la propuesta debe, en lenguaje formal, describir la sintaxis y semántica de la nueva funcionalidad. En otras palabras, se redacta un borrador o primera versión de lo que estará en la especificación. Esta es la etapa en la que se definen realmente todos los aspectos de la característica. Todavía pueden producirse cambios futuros, pero deberían ser menores e incrementales.

Después sigue la Etapa 3. En este punto, la propuesta está casi terminada y solo necesita retroalimentación de implementadores y usuarios para avanzar más. Para progresar a la Etapa 3, el texto de la especificación debe estar terminado y deben existir al menos dos implementaciones compatibles con la especificación.

La última etapa es la Etapa 4. En este punto, la propuesta está lista para ser incluida en la especificación oficial. Para llegar a la Etapa 4, se deben escribir pruebas, dos implementaciones compatibles deben aprobar dichas pruebas, los miembros deben tener experiencia práctica significativa con la nueva característica y el editor de la especificación EcmaScript debe aprobar el texto de la especificación. Básicamente, una vez que una propuesta llega a la Etapa 4, está lista para dejar de ser una propuesta e incorporarse a la especificación oficial. Esto nos lleva a lo último que necesitas saber sobre este proceso: el calendario de lanzamientos del TC39.

Desde 2016, se publica una nueva versión de ECMAScript cada año con las características que estén listas en ese momento. Esto significa que cualquier propuesta en Etapa 4 que exista cuando se lance una nueva versión será incluida en la publicación de ese año. Debido a este ciclo de lanzamientos anual, las nuevas características deberían ser mucho más incrementales y fáciles de adoptar.
