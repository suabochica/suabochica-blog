* Native Features

The last piece of the puzzle is native APIs. We're talking animations, geolocation, notifications, the phone camera — and eventually, shipping to the iOS and Android app stores. Let's go through each one.

** Geolocation

Accessing the user's location is one of the most common native features. Expo gives us a clean JavaScript API that works on both iOS and Android:

#+BEGIN_SRC js
import { Location } from 'expo'
#+END_SRC

When dealing with location services, you need two things:

1. *~getCurrentPositionAsync~*: Gets the current location once, without watching for updates.
2. *~watchPositionAsync~*: Gets the current location /and/ subscribes to updates. The device moves? You'll know.

Check the [[https://docs.expo.io/versions/latest/sdk/location.html][Location docs]] for the full API.

*** Handling Permissions

Whenever a feature needs the user's permission, you need to account for every possible UI state. With location, there are three scenarios:

1. The user grants permission (best case).
2. The user neither denies nor grants permission (the indecisive middle ground).
3. The user flat-out denies access.

In an ideal world, everyone grants permission. But as a UI developer, you plan for all of it. Show different screens, different messages — don't just break when the user says no. ~watchPositionAsync~ is your go-to for continuous tracking; just make sure your UI handles all three permission states gracefully.

** Animations

Animations are a fundamental part of any native app. React Native ships with a built-in animation library called *~Animated~*.

The idea behind ~Animated~ is "declarative relationships between inputs and outputs, with configurable transforms in between, and simple start/stop methods to control time-based animation execution." In plain English: you define transformations on specific values. Animate an image's ~opacity~ from 0 to 1, and it slowly fades in.

There are three animation configurations out of the box:

1. *~decay~*: Start with an initial velocity and gradually slow to a stop.
2. *~spring~*: A normal spring-type animation.
3. *~timing~*: Animate a value over a specified time.

Here's a quick example:

#+BEGIN_SRC js
import React, { Component } from 'react'
import { Animated, Image, StyleSheet, View } from 'react-native'

export default class extends Component {
    state = {
        opacity: new Animated.Value(0),
        height: new Animated.Value(0),
        width: new Animated.Value(0),
    }

    componentDidMount () {
        const { opacity, height, width } = this.state

        Animated.timing(opacity, { toValue: 1, duration: 100 }).start()
        Animated.timing(height, { toValue: 300, speed: 5 }).start()
        Animated.timing(width, { toValue: 300, speed: 5 }).start()
    }

    render () {
        const { opacity, height, width } = this.state

        return (
            <View style={styles.container}>
                <Animated.Image
                    style={[styles.img, { opacity, height, width }]}
                    source={{uri: 'https://www.clavecd.es/wp-content/uploads/buy-the-legend-of-zelda-breathe-of-the-wild-expansion-pass-cd-key-pc-download-img1.jpg'}}
                />
            </View>
        )
    }

    const styles = StyleSheet.create({
        container: {
            alignItems: 'center',
            flex: 1,
            justifyContent: 'center',
        },
        img: {
            height: 200,
            width: 200,
        }
    })
}
#+END_SRC

*** A Word of Caution

Once you grasp the ~Animated~ API, a whole new world opens up. And that's a double-edged sword. You now have the power to enhance your app's feel — but with great power comes great responsibility.

The goal of animations is to /add/ to the user's experience, not distract from it. Keep that in mind and you'll build better, more performant apps. Unnecessary animations ruin UX faster than no animations at all. Use ~decay~, ~spring~, and ~timing~ deliberately, and your app will feel polished rather than chaotic.

** Local Notifications

A notification is a pop-up you get on your phone whether or not the app is running. Expo gives us a ~Notifications~ API with ~scheduleLocalNotificationAsync~ to fire local notifications at specific times.

There are two types of notifications:

1. *Local Notifications*: No external infrastructure needed. Everything happens on the device. The only requirement: the device is on.
2. *Push Notifications*: Requires a server to push notifications to users' devices when specific events occur.

Since we're not setting up a server and all the notification logic can live on the phone, local notifications are the right fit for UdaciFitness. Expo's ~Notifications~ API handles the scheduling; just remember that on iOS, notifications don't appear at the top of the screen if the app is in the foreground, and push notifications won't work in the iOS simulator at all.

** Handling Photos

Expo provides ~ImagePicker~ — a JavaScript abstraction over the native iOS and Android approaches to accessing photos. It does exactly what the name suggests: gives you access to the system's UI for selecting images from the photo library, or taking a photo with the camera. One API, both platforms — that's the Expo promise.

** App Store Preparation

Shipping to either app store requires more than just your app. You need a description, an icon, and a bunch of other metadata. Expo makes this easy — just edit the ~app.json~ file at the root of your project.

*** What are .apk and .ipa files?

Before submitting, you need to /package/ your app:

- iOS App Store wants a *~.ipa~* ("iOS App Store Package") file
- Google Play wants an *~.apk~* ("Android Application Package") file

Each is a bundle of everything the store needs to process and run your app.

The easiest way to generate both is Expo's ~exp~ CLI:

#+BEGIN_SRC bash
npm install -g exp
#+END_SRC

Once installed (and after configuring ~app.json~), run:

#+BEGIN_SRC bash
exp build:ios
exp build:android
#+END_SRC

The React Native server needs to be running when you build.

These builds take 10–20 minutes. Check the status with ~exp build:status~. Eventually you'll get a URL where you can download your ~.ipa~ or ~.apk~ file.

*** The Rest of the Way

The hardest part is generating the ~.ipa~ or ~.apk~ file. The actual upload process is well documented:

- iOS: [[https://developer.apple.com/library/content/documentation/LanguagesUtilities/Conceptual/iTunesConnect_Guide/Chapters/UploadingBinariesforanApp.html#//apple_ref/doc/uid/TP40011225-CH38-SW1][Uploading a Build for an App]]
- Android: [[https://support.google.com/googleplay/android-developer/answer/113469?hl=en][Upload an App]]

** Summary

Expo makes native features accessible through clean JavaScript APIs. For geolocation, ~watchPositionAsync~ tracks the user's movement — just handle all three permission states in your UI. For animations, the ~Animated~ library gives you ~decay~, ~spring~, and ~timing~ — use them deliberately to enhance UX, not distract from it. Local notifications via Expo's ~Notifications~ API work without a server, and ~ImagePicker~ handles photos from the gallery or camera with a single cross-platform API.

When it's time to ship, configure ~app.json~, run ~exp build:ios~ and ~exp build:android~, and Expo generates the ~.ipa~ and ~.apk~ files. From there, follow Apple and Google's upload docs and you're live.
